Frontend Engineering Challenge · Project Report

My Wall Calendar

An interactive, responsive wall calendar component inspired by the tactile charm of a physical paper calendar — built as a frontend engineering challenge submission.

Framework Next.js 15 + React 19
Styling Tailwind CSS v4
Language TypeScript
Storage Client-side React state
Author Aprajita Ranjan

Overview

This project implements a polished, interactive wall calendar component using Next.js and React. The challenge called for a physical-calendar aesthetic with a hero image panel, day-range selection, and an integrated notes section — all fully responsive. Beyond the baseline requirements, the project extends into four visual theme presets, 3D page-flip animations, SVG ring bindings, Indian public holiday markers, and weekend color-coding.

Strictly Frontend All data — including selected date ranges and notes — lives in React state (useState) and is intentionally session-only. No backend, database, or API was built, consistent with the brief's explicit guidance.

Core Requirements Coverage

1 · Wall Calendar Aesthetic

The UI is modelled on a physical desk/wall calendar. Key design elements:

  • A paper-texture background applied via base64 PNG pattern (.paper / .paper-dark CSS classes)
  • A hero image panel at the top that is unique per month and theme (48 unique images across 4 themes × 12 months, all sourced from Unsplash)
  • An SVG wave transition between the hero image and the date grid, using a dynamic linearGradient filled from the active theme's accent colour
  • Ring binding rendered as a row of SVG circles across the card top, simulating the metal rings of a physical calendar
  • Stacked page shadow layers (two absolutely-positioned sibling divs) below the card that simulate the "pages below the current one"
  • A large drop-shadow CSS filter around the entire card assembly

2 · Day Range Selector

The selection model is a two-step click flow managed by the selecting state ("start" | "end"). A live hover preview is maintained in hoverDate state and merged into a previewRange that is passed to buildCalendarDays(), making the range under the cursor visible before the user commits.

Visual State Condition Appearance
Start date selectionState === "start" Accent fill, white text, rounded-left corners
End date selectionState === "end" Accent fill, white text, rounded-right corners
In range selectionState === "in-range" 20% accent tint, no border-radius (flush with neighbours)
Today isToday && selectionState === "none" Accent-coloured outline (2px inset)
Weekend isWeekend && isCurrentMonth Red-tinted date number
Out-of-month !isCurrentMonth 20% opacity

The isBefore guard in handleDayClick ensures that if the user picks an end date earlier than the start, the two dates are automatically swapped — eliminating invalid "backward" ranges.

3 · Integrated Notes Section

The notes panel (NotesPanel.tsx) supports:

  • Month-scoped notes — stored in allNotes: Record<string, Note[]> keyed by "yyyy-MM", so each month has its own independent list
  • Optional date-range association — if a range is active when a note is added, the note stores rangeStart, rangeEnd, and a human-readable dateRangeLabel
  • Range restoration on click — clicking a note that has a saved range re-applies that range to the calendar grid via handleNoteClickAction
  • Delete on hover — a Trash2 icon appears via group-hover:opacity-100, triggered by stopPropagation so the click doesn't also activate range restoration
  • Placeholder text adapts to indicate whether a range is active: "Note for Jan 3 – Jan 8…" vs. "Add a note…"
  • Keyboard shortcut: Enter submits the note input

4 · Fully Responsive Design

Responsiveness is achieved through a combination of Tailwind utility classes and inline style constraints. The calendar renders correctly from ~320px mobile to large desktop displays.

≥ 640px (Desktop)

580px max-width card, centred with top/side padding. Page-flip animation fully visible. ThemeSwitcher pills in a single row.

480–639px (Tablet)

Card fills ~95% viewport width. Ring binding still rendered. Notes panel stacks below grid natively.

< 480px (Mobile)

Full-width card, reduced padding. ThemeSwitcher wraps to two rows. Note input and button stack vertically. Touch-friendly tap targets.

Component Architecture

State Management in CalendarRoot

All state lives in CalendarRoot.tsx, the single source of truth. Child components receive props and callbacks; there is no Context or external store.

State VariableTypePurpose
currentMonthDateThe month currently displayed
rangeDateRangeActive start/end selection
hoverDateDate | nullLive cursor position for range preview
selecting"start" | "end"Which end of the range is being set next
themeNameThemeNameActive visual theme
allNotesRecord<string, Note[]>All notes keyed by "yyyy-MM"
flipDirection"forward" | "backward"Drives the page-flip CSS class

Data Flow

CalendarRoot
├── ThemeSwitcher        ← receives active theme, fires setThemeName
├── PageFlip             ← receives monthKey + direction, wraps children with flip CSS
│   ├── HeroImage        ← receives theme + month metadata, renders Unsplash image + wave
│   ├── [nav controls]   ← inline JSX (today button, prev/next, range label pill)
│   ├── CalendarGrid     ← receives DayInfo[], callbacks for click+hover, theme accent
│   │   └── DayCell[]    ← stateless; renders date, holiday dot, hover tooltip
│   └── NotesPanel       ← receives notes[], label, add/delete/click callbacks, accent

Utility Layer

src/lib/calendarUtils.ts contains:

  • buildCalendarDays(month, range) — generates a 42-cell grid (6 weeks × 7 days) with all DayInfo properties pre-computed, including holiday lookups and weekend flags
  • formatDateRangeLabel(range) — produces human-readable strings like "Jan 3 – Jan 8" or "Apr 1"

Performance Considerations

buildCalendarDays is wrapped in useMemo([currentMonth, previewRange]) so the 42-cell array is only recomputed when the month or range preview changes — not on every keystroke in the notes input. All event handlers in CalendarRoot are wrapped in useCallback to prevent unnecessary re-renders in child components.

Why no framer-motion / three.js? Both framer-motion and three are in package.json as dependencies but are not used in the final implementation. The page-flip animation was achievable with a single CSS @keyframes rule and a conditional class name — adding a 40KB+ animation library for this one interaction was unnecessary. The physical paper aesthetic similarly didn't require WebGL.

Responsive Design Notes

The calendar intentionally uses a single-column card layout rather than a multi-panel desktop layout. This decision mirrors the reference image (a compact portrait-format wall calendar) and means the same layout works from mobile upward with minimal breakpoint logic.

Touch Interaction

No special touch handling was required. All interactive elements use standard onClick and onMouseEnter handlers — React's synthetic event system already normalises these for touch on mobile. Day cells are h-12 (48px) tall, meeting the 44px minimum touch target recommendation.

Code Quality & Architecture Decisions

TypeScript Coverage

All components, utility functions, and state variables are typed. Key shared types in src/types/calendar.ts:

DayInfo Note DateRange ThemeName MonthTheme SelectionState

Component Responsibilities

Components follow a strict separation of concerns:

  • CalendarRoot — state, business logic, callback wiring. Zero JSX layout involvement beyond structural wrappers.
  • DayCell — purely presentational; receives all props, returns UI. No internal state.
  • CalendarGrid — maps DayInfo[] → DayCell[]. Single responsibility.
  • PageFlip — animation wrapper only; knows nothing about calendar data.
  • HeroImage — renders image + wave SVG. Receives theme object and month metadata.

Naming Conventions

Event handler props follow the onXxxAction pattern (e.g., onAddAction, onDeleteAction, onNoteClickAction) to disambiguate prop callbacks from native DOM events in TypeScript autocomplete.

CSS Architecture

Structural layout uses Tailwind utilities. Component-specific or complex styles (paper textures, keyframe animations, page-flip, perspective transforms) live in globals.css as named classes. Inline styles are used only for dynamic theme values that change at runtime (accent colours, background colours).
